lesson 01

What is a Builder's Mindset?

Before you write a single line of code, you need to understand how builders think. It's a specific way of looking at the world — and you can learn it.

THE IDEA

A builder doesn't see a broken website and think "that's someone else's problem." They think "I wonder how I'd fix that." A builder doesn't see a tedious task and just accept it — they think "I could probably automate this."

This isn't a skill you're born with. It's a habit of mind. And like any habit, you build it through practice — by repeatedly asking better questions about the world around you.

The best developers, designers, and freelancers in the world share one thing: they treat every problem as a puzzle to be broken apart, not a wall to be stopped by.

THE THREE QUESTIONS EVERY BUILDER ASKS

1. What is actually happening here? (Understand the real problem, not just the symptom)
2. What would need to be true for this to work? (Think backwards from the solution)
3. What's the smallest version of this I could test? (Start tiny, iterate fast)

THE DESIGN PROCESS CONNECTION

If you've done design — or art — you already know this process. You just didn't call it programming. In design you: research → define → ideate → prototype → test.

In software, the exact same process applies. Every app, every website, every AI tool was built by someone who went through exactly those steps. The tools are different. The thinking is identical.

This course uses the design process as a framework throughout — because it's the best mental model for building anything.

lesson 02

Patterns Everywhere

Maths isn't about numbers. It's about recognising patterns. And pattern recognition is one of the most powerful skills a developer — and a freelancer — can have.

THE REAL POINT OF MATHS

Most people think maths is about calculating answers. It's not. Maths is about seeing structure in chaos. When you spot that something repeats, scales, or follows a rule — that's mathematical thinking. And it shows up everywhere in software.

A loop in code is a pattern. A grid layout on a website is a pattern. A pricing model is a pattern. The way a virus spreads — a pattern. When you learn to spot the underlying structure of something, you can predict it, automate it, and build with it.

This is also how AI works. Language models like Claude don't "understand" language — they detect and reproduce patterns at massive scale. Knowing this makes you a much better prompt writer.

PATTERN RECOGNITION IN REAL PROJECTS

When a client says "I want users to be able to log in" — that's a pattern you've seen before (authentication). When they say "I want it to update in real time" — that's another pattern (websockets or polling). Experienced developers recognise patterns faster, which is why they build faster.

THE ART CONNECTION

Art and maths have more in common than people think. A grid in design is a mathematical system. Colour theory is based on wavelength ratios. A well-designed logo uses geometric proportions. Visual rhythm in a layout is pattern. If you studied design or art, you've been doing mathematical thinking all along.

This is your advantage. You can see how things should be structured — visually and logically. That's exactly what good UI, good code architecture, and good client communication all require.

lesson 03

Design Thinking Meets Code

The design process you learned in school is the same process professional developers use. Here's how to apply it to every project you'll ever build.

YOU ALREADY KNOW THIS

If you've done any design subject, you've been trained to solve problems in a structured way. You research, define, ideate, prototype, test. You iterate. You get feedback. You revise. This is exactly how good software is built.

Most beginner developers skip straight to building. They start typing code before they've understood the problem. This is why so many projects fail — not because the code was bad, but because the thinking that came before it was skipped.

As a freelancer, the person who asks the best questions before starting wins. Clients don't always know what they want — your job is to find out.

THE FREELANCER'S VERSION OF THE DESIGN PROCESS

1. Discovery call — ask, listen, understand what the client actually needs
2. Project brief — write it back to them in your own words
3. Approach options — offer 2–3 ways you could solve it
4. Build the MVP — the smallest version that proves the concept
5. Review + iterate — show it, get feedback, refine

lesson 04

Breaking Problems Into Pieces

Decomposition — breaking a big problem into small ones — is the single most important skill in software development. Here's how to do it.

THE SKILL BEHIND THE SKILL

When beginners see a big project, they freeze. "I don't even know where to start." When experienced developers see the same project, they immediately start breaking it apart. Not because they know more code — because they know how to decompose.

Decomposition is simple: take the big thing, and keep asking "what are the parts of this?" until each part is small enough to actually do. Then start with the smallest piece. Build it. It works. Now you have momentum.

EXAMPLE: "I WANT TO BUILD A WEBSITE"

Website → Pages → Header, Body, Footer
Header → Logo, Navigation links, Button
Navigation → List of links → Each link → URL + text

Now you have something you can actually start: write one navigation link. That's it. One link. Then another. Then the list. Then the header. Then the page. That's how websites get built.

THIS IS ALSO HOW YOU PROMPT AI

When you write a prompt to Claude or ChatGPT asking it to "build me a website", you'll get something generic. But when you decompose first, your prompt becomes specific: "Create a navigation bar with these 4 links, using this colour, with a hamburger menu on mobile." That gets you something useful.

Decomposition makes your AI 10x more powerful. The better you can break down what you want, the better your AI output will be. This is a skill most people never develop — which means you'll be ahead of nearly everyone.

ACTIVITY — 25 MINUTES
STEP 01

Decompose a real app

Pick an app you use daily (Spotify, Instagram, Google Maps). In your workbook, draw a tree diagram breaking it into parts, sub-parts, and smallest pieces. Go at least 3 levels deep.

// 15 min
STEP 02

Write a micro-prompt

Take the smallest piece from your diagram. Write a specific, detailed AI prompt that could help you build just that piece. No vague language allowed — be exact.

// 10 min

Reflect

When was the last time you felt overwhelmed by something? Could you have broken it down further?

Why does a specific prompt get better AI results than a vague one?

lesson 05

Your First AI Conversation

Time to use Claude for real. You'll learn how to have a productive conversation with AI — not just ask it questions, but guide it like a collaborator.

AI IS A COLLABORATOR, NOT A SEARCH ENGINE

Most people use AI like Google — type a question, read the answer, close the tab. That's wasting most of its value. AI is at its most powerful when you treat it like a very capable colleague who needs context and direction.

Good AI prompting has a structure: context → task → constraints → format. You tell it who you are and what you're doing, what you want it to do, what rules to follow, and how you want the answer presented. That combination unlocks dramatically better results.

THE PROMPT FORMULA

Context: "I'm a high school student building my first website..."
Task: "...create a simple navigation bar in HTML..."
Constraints: "...with no external libraries, using only plain CSS..."
Format: "...and explain each line in plain English." 

// WEAK PROMPT (gets generic results)
"Make me a website"

// STRONG PROMPT (gets useful results)
"I'm learning HTML for the first time.
Create a navigation bar with 4 links:
Home, About, Projects, Contact.
Use plain HTML and CSS only — no frameworks.
Make it responsive on mobile.
Explain what each CSS property does in a comment."

lesson 06

The Freelance Mindset

Everything you build in StackUp is pointing toward one goal: the ability to earn income with your skills. Here's what that actually looks like — and how to start before you even finish school.

WHAT FREELANCING ACTUALLY IS

Freelancing means solving real problems for real people and getting paid for it. Not having a boss. Not waiting until you're 22 and have a degree. You can start taking small clients the moment you have a useful skill.

Australian businesses — especially small ones — need websites, automation, and digital tools. Most of them don't have the budget for a big agency. They need someone who is affordable, communicates well, and delivers. That's you.

Your first freelance project doesn't have to be impressive. It just has to solve a real problem for a real person. That's the whole game.

THE FREELANCER'S PATH (REALISTIC VERSION)

Month 1–3: Build skills + 2 practice projects (no client yet)
Month 3–5: First project for someone you know — cheap or free to get feedback
Month 5–8: First paid client — small scope, simple delivery
Month 8+: Referrals, case studies, portfolio, raise your rates

SHADOWING A FREELANCER

In StackUp, top students get connected to working freelancers — not big companies, but real independent developers who do exactly what you'll eventually do. You sit alongside them (virtually), see how they find clients, how they scope work, how they handle problems, and how they get paid.

This is worth more than any textbook. Seeing someone do something makes it real in a way that reading about it never does.
